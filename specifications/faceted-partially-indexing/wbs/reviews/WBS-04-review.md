# WBS-04 Review: Catalog Expression Trigger Registry

## Verdict: PASS WITH NOTES

## Summary

This is an exceptionally thorough WBS document with deep source code research, well-structured task decomposition, and comprehensive test coverage. The author has verified class locations, line numbers, call chains, and existing patterns in the codebase, producing a document that a developer can follow confidently. There are a few design tensions (the `rebuildForEntityType()` interface method vs. static factory approach), one inconsistency with the existing `removeLayer()` pattern, and a handful of missing edge cases in tests and tasks, but none are blocking.

## Issues Found

### Critical (must fix before implementation)

None.

### Important (should fix)

1. **`rebuildForEntityType()` interface method vs. static factory design tension (Tasks 3.1 and 3.2)** -- The parent analysis defines `rebuildForEntityType(String entityType)` as a method on the `CatalogExpressionTriggerRegistry` interface, returning a new registry instance. However, task 3.1 says the method "should NOT internally access the catalog" and should be a "pure function," and task 3.2 introduces a separate static factory `CatalogExpressionTriggerRegistryImpl.rebuildForEntityType(current, entityType, newTriggers)` with different parameters. This leaves the interface method's implementation unclear -- if the interface method takes only `entityType`, how does the implementation get the new triggers? Two possibilities: (a) the interface method signature is updated to accept `List<ExpressionIndexTrigger>` as an additional parameter, or (b) the interface method delegates to the static factory internally, but then the caller (Catalog) needs to build the triggers first and somehow pass them. Task 7.1 shows the Catalog doing the orchestration and calling the static factory directly, which suggests the interface method `rebuildForEntityType(String)` may never actually be called as-is. The WBS should clarify whether the interface method signature needs to change or whether `Catalog` bypasses the interface method entirely and uses the static factory. If the latter, the interface method definition copied from the parent analysis is misleading.

> ✅ **Addressed**: Resolved by updating the interface method signature to `rebuildForEntityType(String entityType, List<ExpressionIndexTrigger> newTriggers)`, making it a pure function that receives the trigger list from the caller. The static factory in Task 3.2 was removed in favor of making the interface method itself the rebuild entry point. Tasks 3.1, 3.2, 7.1, 7.2, and the Scope, Acceptance Criteria, and Implementation Notes sections were all updated for consistency. The Catalog orchestrates trigger building via factories and passes the result to the interface method.

2. **`removeLayer()` pattern inconsistency (Task 6.4)** -- Task 6.4 says to add `this.expressionTriggerRegistry.removeLayer(transactionalLayer)` to `Catalog.removeLayer()`, "following the pattern of all other `TransactionalReference`/`TransactionalMap` fields." However, the existing `versionId` field is also a `TransactionalReference<Long>` (line 185 of `Catalog.java`) and is NOT included in `removeLayer()` (lines 1432-1442). The WBS should either acknowledge this discrepancy and explain why the registry should be in `removeLayer()` even though `versionId` is not, or investigate whether `versionId` is missing from `removeLayer()` by design (perhaps it is never transactionally modified in a way that requires layer cleanup). Adding the registry to `removeLayer()` is likely correct, but the stated rationale ("follows the pattern of all other fields") is inaccurate.

> ✅ **Addressed**: Updated Task 6.4 and the source research section to acknowledge that `versionId` is excluded from `removeLayer()` (likely by design, since it is managed via `setVersion()` and merged separately in `createCopyWithMergedTransactionalMemory()`). The rationale now correctly references the `schema` field (also a `TransactionalReference`) as the analogous pattern, and explains why the registry needs layer cleanup (it can be set within a transaction and must be cleaned on rollback).

3. **Missing handling for entity type as mutated entity (self-referencing expressions) (Tasks 3.3, 4.1)** -- The WBS assumes the mutated entity type is always different from the owner entity type (Product references ParameterGroup). But what if an entity has a self-referencing reference (e.g., Product references Product as group)? In this case, `getReferencedGroupType()` returns the same entity type as the owner. The rebuild for "product" must then handle triggers indexed under "product" owned by "product" without accidentally cleaning them during the stale-trigger-removal step (task 3.2 step 2 removes all entries where any trigger has `getOwnerEntityType().equals(entityType)` -- but if the owner is also the mutated entity type key, this could remove the very triggers being inserted). The WBS should add a note about this edge case and a test case.

> ✅ **Addressed**: Added Task 3.4 documenting the self-referencing edge case. Analysis shows the rebuild algorithm actually handles this correctly: step 1 removes old triggers by owner, step 3 inserts new triggers -- the operations are sequential, so removing old triggers before inserting new ones works even when the owner type equals the mutated entity type key. Added a dedicated test case `rebuildForEntityType_self_referencing_entity_type_handled_correctly` in Category 6 to explicitly verify this.

4. **`renameCollection` trigger handling is underspecified (Task 9.2)** -- Task 9.2 says "renaming entity type from 'product' to 'item' results in triggers previously owned by 'product' now being owned by 'item'." However, the triggers stored in the registry are immutable objects with `getOwnerEntityType()` baked in. The rename cannot just move triggers -- it must rebuild them with the new owner entity type. Furthermore, if entity A references entity B under B's old name, the reference schema's `getReferencedEntityType()` / `getReferencedGroupType()` may still have the old name until the schema cascade propagates. The WBS should specify whether the rename causes a full registry rebuild or a targeted trigger reconstruction, and how it interacts with schema cascade timing. The test case in Category 12 (`renameCollection_reindexes_triggers_under_new_name`) asserts "triggers previously owned by 'product' now being owned by 'item'" which implies trigger reconstruction, but no task describes how to achieve this.

> ✅ **Addressed**: Task 9.2 was substantially expanded to specify the complete rename flow: (1) old triggers are explicitly cleaned via `rebuildForEntityType(oldName, emptyList)`, (2) new triggers are built from the updated schema with the new owner entity type, (3) `notifyEntityTypeRenamed()` cascades to other collections whose reference schemas point to the renamed entity, triggering their own registry rebuilds with updated mutated entity type keys. The test case was also updated to verify the two-step clean-then-rebuild process.

5. **AD 19 reference may confuse readers (Technical Context section)** -- The "Immutability Principle" subsection states "This is captured in AD 19 of the parent analysis document." AD 19 does exist in the parent analysis, but it is numbered within a long numbered list without explicit "AD 19" labeling -- it is just item 19 in the "Architecture Decisions / Decided" section. A developer looking for "AD 19" as a heading would not find it. Consider referencing it as "Architecture Decision item 19" or linking to the section heading.

> ✅ **Addressed**: Updated the reference to read "Architecture Decision item 19 ('CatalogExpressionTriggerRegistry immutability') in the parent analysis document's 'Architecture Decisions / Decided' numbered list."

### Minor (nice to have)

1. **`buildFromSchemas()` does not account for future histogram triggers (Task 4.1)** -- Task 4.1 only calls `FacetExpressionTriggerFactory.buildTriggersForReference()`. The WBS scope says the registry is generic and stores all `ExpressionIndexTrigger` subtypes, but the cold start build only processes facet triggers. When `HistogramExpressionTrigger` support is added, this method will need extension. A brief note acknowledging this extensibility point would be helpful, even if histogram triggers are out of scope.

> ✅ **Addressed**: Added an extensibility note to Task 4.1 explaining that when `HistogramExpressionTrigger` support is added, `buildFromSchemas()` and `Catalog.rebuildExpressionTriggerRegistryForEntityType()` must be extended with an additional factory call. Notes that the registry data structure already supports this with no structural changes needed.

2. **No test case for scope isolation** -- The registry stores triggers with `Scope` metadata (a reference with expressions in both `LIVE` and `ARCHIVED` scopes produces separate triggers). While Category 7 (`buildFromSchemas_reference_with_expressions_in_both_scopes_produces_two_triggers`) covers construction, there is no test verifying that `getTriggersFor()` or `getTriggersForAttribute()` returns triggers for BOTH scopes when a reference has expressions in multiple scopes. The scope is on the trigger object, not a registry key dimension, so both triggers should be in the same list -- but this should be verified explicitly.

> ✅ **Addressed**: Extended the existing test case `buildFromSchemas_reference_with_expressions_in_both_scopes_produces_two_triggers` to also verify that both `getTriggersFor()` and `getTriggersForAttribute()` return both scope-specific triggers in the same result list.

3. **No test for concurrent rebuild + read interaction via `TransactionalReference` (Category 14)** -- Category 14 tests `TransactionalReference` set/get behavior, and Category 13 tests concurrent reads on a bare registry instance. However, there is no test that combines both: thread A calls `Catalog.rebuildExpressionTriggerRegistryForEntityType()` (which sets the `TransactionalReference`) while threads B and C read via `Catalog.getExpressionTriggerRegistry()`. This would verify the complete thread-safety story end-to-end through the `TransactionalReference` wrapper.

> ✅ **Addressed**: Added test case `concurrent_rebuild_via_transactional_reference_and_read` in Category 13 that combines both: thread A rebuilds via `Catalog.rebuildExpressionTriggerRegistryForEntityType()` while threads B and C read via `catalog.getExpressionTriggerRegistry()`, verifying transactional isolation end-to-end.

4. **Line number references may drift** -- The source code research section references specific line numbers in `Catalog.java` (e.g., "line 179-180", "line 194", "lines 488-498", "lines 711-784", "line 1409-1411", "line 1432-1442"). These are accurate as of the current codebase state but will drift as other WBS items are implemented. This is not a defect, just a maintenance note. Consider anchoring references to method names or code patterns instead of (or in addition to) line numbers where possible.

> ✅ **Addressed**: Added a note at the top of the Source Code Research Results section warning that line numbers may drift and advising to locate code by method name or code pattern when they no longer match.

5. **`getTriggersForAttribute()` allocation optimization not fully specified (Task 2.4)** -- Task 2.4 says to use "allocation-optimized loop per code-style rules (avoid streams)" but does not specify whether the filtered list should be pre-allocated (e.g., `new ArrayList<>(triggers.size())`) or use a collector pattern. For a performance-critical lookup path, the WBS could specify to first count matches (or just accept the small overhead of an ArrayList with initial capacity equal to the source list size).

> ✅ **Addressed**: Updated Task 2.4 to specify pre-allocating the result list with `new ArrayList<>(triggers.size())` as initial capacity, noting the small over-allocation is acceptable.

6. **`CatalogRelatedDataStructure` not used** -- The source code research results table includes `CatalogRelatedDataStructure` but it is never referenced in any task or implementation note. If it was considered and rejected as a pattern for the registry, a brief note would be useful. If it was included speculatively, it can be removed to reduce noise.

> ✅ **Addressed**: Added a note after the key class locations table explaining that `CatalogRelatedDataStructure` was investigated but rejected -- it is designed for data structures needing late initialization and catalog attachment propagation to children, while the registry is a simpler plain immutable POJO wrapped in `TransactionalReference` with no children.

7. **Empty `dependentAttributes` trigger behavior is tested but not explicitly specified in tasks** -- Test case `getTriggersForAttribute_with_trigger_having_empty_dependent_attributes` asserts that a trigger with an empty `dependentAttributes` set is never returned by `getTriggersForAttribute()`. This is a consequence of `Set.contains()` on an empty set returning false, but the behavior is not explicitly stated in any task or acceptance criterion. Consider adding a brief note in the implementation notes section.

> ✅ **Addressed**: Added a behavioral note in the Implementation Notes section explaining that triggers with empty `dependentAttributes` are never returned by `getTriggersForAttribute()` (since `Set.contains()` on an empty set returns false), and that this is the correct behavior for triggers with no cross-entity attribute dependencies.
