# WBS-08 Review: Source-Side Detection

## Verdict: PASS WITH NOTES

## Summary

WBS-08 is a well-structured, thorough specification for the source-side detection logic. The
document demonstrates deep knowledge of the existing codebase with accurate line-number
references, a clear 5-step pipeline, and comprehensive test coverage. There are a few scope
overlap issues with WBS-05 and WBS-10, one inconsistency in how old values are determined for
the old-vs-new comparison, and a missing edge case around localized attributes, but none of
these are blocking.

## Issues Found

### Critical (must fix before implementation)

None.

### Important (should fix)

- **Scope overlap with WBS-10 for dispatch integration.** WBS-08 In Scope states "Dispatch
  integration in `LocalMutationExecutorCollector` (Step 5b)" and Group 5 (tasks 5.1-5.4)
  specifies the full dispatch loop implementation. However, WBS-10 is titled
  "`LocalMutationExecutorCollector` Integration -- Two-Loop Dispatch" and claims this exact
  same work in its own scope. WBS-08 even lists WBS-10 as "Depended On By" (line 55-56), yet
  simultaneously implements the dispatch itself. Either WBS-08 Group 5 should be removed (and
  left entirely to WBS-10), or WBS-10's scope should be narrowed to only the
  `applyIndexMutations()` method and `IndexMutationTarget` wiring on `EntityCollection`, with
  the collector dispatch loop clearly owned by WBS-08. As it stands, both WBS documents have
  tasks to add the same code block to `LocalMutationExecutorCollector.execute()`.

> ✅ **Addressed**: Narrowed WBS-08 scope — In Scope now says "Defining the dispatch contract" instead of "Dispatch integration." Group 5 was rewritten to document the contract and expected dispatch behavior for WBS-10 consumption, not to implement the dispatch loop itself. Tasks 5.1-5.4 now verify the output contract rather than adding code to `LocalMutationExecutorCollector.execute()`. Task 5.2 explicitly references WBS-10 as the owner of `applyIndexMutations()`. Acceptance criterion 8 was also updated to describe the output contract rather than the dispatch implementation.

- **`IndexImplicitMutations` record ownership ambiguity.** WBS-08 Group 3 task 3.1 specifies
  creating the `IndexImplicitMutations` record as a nested type in
  `EntityIndexLocalMutationExecutor`. However, WBS-05 (Mutation Type Hierarchy) also claims
  `IndexImplicitMutations` in its scope (line 25: "New `IndexImplicitMutations` record for
  `EntityIndexLocalMutationExecutor` output") and has its own Group 6 task for creating it.
  One of these must be designated as the canonical owner to avoid duplicate work or merge
  conflicts.

> ✅ **Addressed**: WBS-05 is now designated as the canonical owner. Task 3.1 was rewritten to state that the record and stub method are created by WBS-05 (Group 6), and WBS-08 assumes they already exist. The In Scope bullet for `IndexImplicitMutations` now clarifies "(record definition owned by WBS-05; this WBS implements the logic that populates it)."

- **New-value retrieval in step 3.3a is more complex than described.** Task 3.3a says to
  "scan `inputMutations` for the matching `AttributeMutation`" to get the new value. However,
  `inputMutations` may also contain `ReferenceAttributeMutation` wrapped inside
  `ReferenceMutation` (which is a separate branch in `applyMutation()`). Additionally, for
  `EntityRemoveMutation`, the `localMutations` list is generated by
  `computeLocalMutationsForEntityRemoval()` and contains `RemoveAttributeMutation` instances,
  but the executor receives `inputMutations` as a parameter -- the document does not clarify
  whether `inputMutations` in the removal case is the computed list or the original mutation
  list. Looking at `LocalMutationExecutorCollector.execute()` line 227, for removals,
  `localMutations` is set to the computed list from `computeLocalMutationsForEntityRemoval()`.
  The document should clarify this flow more explicitly, especially since task 4.1 uses the
  `isEntityRemovedEntirely()` check as a separate branch that bypasses the attribute scanning
  entirely. If the removal branch bypasses attribute scanning, then the new-value retrieval in
  3.3a never runs for removals, which is correct -- but the relationship between the two
  branches should be stated explicitly.

> ✅ **Addressed**: Three clarifications added. (1) The removal-vs-attribute branch in step 3.2 task 4 now explicitly states the branches are "mutually exclusive" and explains that the removal path bypasses per-attribute scanning entirely. (2) Task 3.3a now clarifies that `ReferenceAttributeMutation` is wrapped in `ReferenceMutation` and falls under a different branch in `applyMutation()`, so it never appears in `cachedOldAttributeValues`. (3) Task 3.3a now states that for `EntityRemoveMutation` the code path never runs because the removal branch handles that case.

- **Localized attributes are not addressed.** The parent analysis (Domain 2b/2c) mentions
  `$reference.referencedEntity.attributes['x']` and
  `$reference.groupEntity?.attributes['x']` but expressions can also reference localized
  attributes (`$entity.localizedAttributes['x']`). The `AttributeKey` includes a `Locale`
  component. The old-value caching in Group 2 uses `AttributeKey` as the map key, which is
  correct, but the registry consultation in step 3.3c uses `attributeKey.attributeName()` --
  discarding the locale. This means if the same attribute name has different values in
  different locales, a change in locale `en` would also fire triggers whose expressions only
  reference locale `cs`. This is not incorrect (it just produces unnecessary trigger firings
  that the target-side handles idempotently), but the document should acknowledge this
  over-firing behavior and explain why it is acceptable (over-firing is safe due to
  idempotency; precise per-locale filtering would require the trigger to declare which locales
  it reads, which is complexity not warranted at this stage).

> ✅ **Addressed**: Added a "Localized attribute over-firing" implementation note in the Implementation Notes section. The note acknowledges that `attributeKey.attributeName()` discards the locale, explains that this is intentional over-firing, and clarifies why it is safe (target-side idempotency) and why per-locale filtering is not warranted at this stage. Also notes that the old-value cache uses the full `AttributeKey` (including locale) so the old-vs-new comparison is locale-correct.

### Minor (nice to have)

- **Task 2.4 sentinel complexity may be unnecessary.** The task proposes a sentinel object to
  distinguish "attribute did not exist" from "attribute exists with null value." However,
  evitaDB attributes are always non-null `Serializable` values (the schema enforces this --
  `AttributeValue.value()` is `@Nonnull Serializable`). A null return from
  `getAttributeValue()` means the attribute did not exist, and `RemoveAttributeMutation`
  results in absence. Therefore, `null` in the map reliably means "did not exist before," and
  no sentinel is needed. The task should note this simplification opportunity.

> ❌ **Declined**: The review's premise is incorrect. `AttributeValue.value()` is annotated `@Nullable Serializable` in the record definition (`AttributesContract.java:392`), not `@Nonnull`. While `ExistingAttributeValueSupplier.getAttributeValue()` does filter dropped attributes via `Droppable::exists`, the `@Nullable` annotation means the sentinel concern is technically valid. However, a simplification note was added to task 2.4 acknowledging that in practice non-dropped attributes typically have non-null values, and implementers may simplify if analysis confirms this.

- **Task 5.2 mentions creating a stub `applyIndexMutations()` on `EntityCollection`.** This
  is reasonable for compilation, but WBS-10 claims ownership of this method. The task should
  reference WBS-10 explicitly and note that the stub is temporary.

> ✅ **Addressed**: Task 5.2 now explicitly references WBS-10 as the owner of `applyIndexMutations()` and the `IndexMutationTarget` interface. The task states that if compilation requires a call target, WBS-10 (not WBS-08) should provide the stub.

- **Task 7.1 edge case handling could be simpler.** The task describes scanning
  `inputMutations` in forward order to find the LAST `AttributeMutation` per key for the new
  value. An alternative (and simpler) approach: after all mutations are applied, read the
  current value from the storage part accessor (which now has the post-mutation value). This
  avoids having to re-scan the mutation list. The document already notes that by the time
  `popIndexImplicitMutations()` runs, storage has the new values. This approach would
  eliminate the need for task 3.3a's mutation scanning entirely.

> ✅ **Addressed**: Added an "Alternative approach" note to task 7.1 describing the post-mutation storage read approach. The note explains that reading from `getStoragePartExistingDataFactory().getEntityAttributeValueSupplier().getAttributeValue(attributeKey)` after all mutations are applied would return the new value, eliminating the need for mutation list scanning and naturally handling the multiple-mutations-per-key case. The implementer is advised to evaluate which approach is simpler.

- **Test location inconsistency.** The test class is specified at
  `evita_test/evita_functional_tests/src/test/java/io/evitadb/index/mutation/index/EntityIndexLocalMutationExecutorTriggerTest.java`.
  The existing `EntityIndexLocalMutationExecutor` tests should be verified to confirm they
  live in the same package/module. This is a minor concern but could cause confusion if
  existing tests use a different base class or test infrastructure.

> ❌ **Declined**: Verified that existing tests for `EntityIndexLocalMutationExecutor` (e.g., `AbstractMutatorTestBase.java`, `AttributeIndexMutatorTest.java`) live in `evita_test/evita_functional_tests/src/test/java/io/evitadb/index/mutation/index/` — the exact same package and module as the proposed test class. No inconsistency exists.

- **Missing test for `ApplyDeltaAttributeMutation` with old==new optimization edge case.**
  Category 2 tests cover the `UpsertAttributeMutation` and `RemoveAttributeMutation` old==new
  cases, but there is no test confirming that `ApplyDeltaAttributeMutation` with a delta of 0
  is handled correctly. Task 3.3a states "any delta implies a change (skip the old==new
  check)" -- but a delta of 0 does NOT actually change the value. A test should verify
  whether delta=0 fires or skips the trigger (and the document should state the intended
  behavior).

> ✅ **Addressed**: Added test case `pop_index_implicit_mutations_fires_for_zero_delta_attribute_mutation` in Category 1 (after the existing delta test). The test explicitly documents the design decision: delta=0 fires the trigger (safe over-firing) because the implementation treats ANY `ApplyDeltaAttributeMutation` as an implicit change to avoid the complexity of computing the absolute new value. The Technical Context section about `ApplyDeltaAttributeMutation` was also updated to mention "including delta=0."

- **WBS-09 is listed as "Depended On By" (line 54) but WBS-09 is about local trigger
  integration in `ReferenceIndexMutator`, which does not consume `EntityIndexMutation`
  envelopes.** WBS-09's dependencies list only WBS-03. The dependency graph should be
  corrected: WBS-10 depends on WBS-08 (correct), but WBS-09 does not appear to depend on
  WBS-08 at all.

> ✅ **Addressed**: Removed WBS-09 from the "Depended On By" section. WBS-09 is about local trigger integration in `ReferenceIndexMutator` and does not consume `EntityIndexMutation` envelopes — confirmed by verifying WBS-09's "Depends On" lists only WBS-03. Additionally, the Out of Scope references to "WBS-09" for target-side execution were corrected to reference WBS-07 (`ReevaluateFacetExpressionExecutor`), which is the actual target-side executor WBS.
