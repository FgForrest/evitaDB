# WBS-02 Review: Proxycian Proxy Infrastructure

## Verdict: PASS WITH NOTES

## Summary

The WBS document is thorough, well-structured, and provides exceptional detail for a developer to implement the Proxycian proxy infrastructure. The task breakdown is logically ordered, references to existing codebase patterns are accurate, and test coverage is comprehensive. However, there are a few technical inconsistencies (attribute lookup strategy vs. existing code, interface signature mismatches, `SealedEntity` interface complexity) and some minor gaps in edge case coverage that should be addressed before implementation begins.

## Issues Found

### Critical (must fix before implementation)

- **`SealedEntity` extends `SealedInstance` which requires `openForEdit()` / `toMutation()`** (Task 7.3): The WBS correctly identifies that nested entity proxies must implement `SealedEntity` (since `ReferenceContract.getReferencedEntity()` and `getGroupEntity()` return `Optional<SealedEntity>`). However, `SealedEntity extends EntityContract, SealedInstance<SealedEntity, EntityBuilder>`, and `SealedInstance` declares an `openForEdit()` method returning an `EntityBuilder`. The `CatchAllPartial` would handle this, but the WBS does not acknowledge the `SealedInstance` contract surface or explicitly confirm that `openForEdit()` should throw `ExpressionEvaluationException`. This could surprise the implementer. Additionally, `EntityContract` extends `ContentComparator<EntityContract>` which adds `differsFrom()` — another method that needs handling. The WBS should explicitly list the full interface surface that `CatchAllPartial` must absorb for `SealedEntity` proxies, or at minimum call out this risk.

> ✅ **Addressed**: Task 7.3 now explicitly lists the three `SealedInstance` abstract methods (`openForWrite()`, `withMutations(LocalMutation...)`, `withMutations(Collection<LocalMutation>)`) that must be absorbed by `CatchAllPartial`. Also clarifies that `differsFrom()` and `estimateSize()` are default methods on `EntityContract` and do not require explicit handling. Note: the actual method name is `openForWrite()`, not `openForEdit()` as stated in the review.

- **`ReferenceProxyState.referencedEntity` and `groupEntity` typed as `SealedEntity`, but Task 6.1 returns `EntityContract`** (Tasks 1.2 vs 6.1): In Task 1.2, `ReferenceProxyState` declares fields `referencedEntity` and `groupEntity` as `SealedEntity`. But in Task 6.1, `ExpressionProxyInstantiator.createProxies()` returns a record with `EntityContract entityProxy` and `ReferenceContract referenceProxy`. The nested entity proxy created in Task 7.1/7.2 must be a `SealedEntity` (per Task 7.3), so the `ExpressionProxyInstantiator` should also produce `SealedEntity` instances for nested proxies, not `EntityContract`. The method signature in Task 6.1 should be updated to reflect this, or at minimum the document should clarify that nested entity proxies are upcast to `SealedEntity` before being stored in `ReferenceProxyState`.

> ✅ **Addressed**: Task 6.1 now includes a detailed note clarifying that the root `entityProxy` is typed as `EntityContract`, while nested entity proxies are `SealedEntity` instances. The note explains that the upcast is guaranteed because nested proxy classes include `SealedEntity.class` in their interfaces array, and that these `SealedEntity` instances are stored in `ReferenceProxyState.referencedEntity` / `groupEntity` fields.

### Important (should fix)

- **Linear scan vs binary search contradiction** (Task 2.3 and Implementation Notes): The WBS repeatedly states that `getAttribute(String)` should use a "linear scan of `AttributeValue[]`" and that this is "faster than HashMap lookup for typical attribute counts (<30 entries) due to cache locality." However, the actual `AttributesStoragePart.findAttribute(AttributeKey)` already provides a **binary search** on the sorted `AttributeValue[]` array (via `Arrays.binarySearch()`). The WBS should either (a) clarify that the partial should use the existing `findAttribute()` method on the storage part (binary search), or (b) explain why a new linear scan implementation is preferred over the existing binary search. Binary search is O(log n) vs O(n) and is already implemented and tested. The performance test in `ExpressionProxyPerformanceTest` (`linear_scan_is_faster_than_hashmap_for_small_attribute_counts`) also benchmarks a linear scan approach that may not match the final implementation.

> ✅ **Addressed**: All references to "linear scan" for attribute lookup have been replaced with delegation to `AttributesStoragePart.findAttribute(AttributeKey)` (binary search). Updated locations: implementation notes (EntityContract partials section), Task 2.3, Task 3.2, Implementation Notes section, Acceptance Criteria item 12, the `EntityAttributePartialTest` test case, and the `ExpressionProxyPerformanceTest` benchmark test case.

- **`createEvaluationContext` signature mismatch with parent analysis** (Key Interfaces section): The WBS defines `createEvaluationContext` as taking `ReferenceContract reference`, but the parent analysis document defines `ExpressionIndexTrigger.evaluate()` as taking `ReferenceKey referenceKey` (not `ReferenceContract`). The WBS positions `createEvaluationContext` on `FacetExpressionTrigger`, while the parent analysis shows `evaluate()` on `ExpressionIndexTrigger`. These are different methods, but the relationship between them is not clearly established. The WBS should clarify: does `evaluate()` call `createEvaluationContext()` internally? If so, where does the `ReferenceContract` come from when `evaluate()` only receives a `ReferenceKey`? The partial implementations (especially `ReferenceAttributePartial`) need the reference's `AttributeValue[]` data, which suggests the reference data must be fetched from `ReferencesStoragePart` using the `ReferenceKey`.

> ✅ **Addressed**: Updated `createEvaluationContext()` signature to take `ReferenceKey referenceKey` (not `ReferenceContract`) in the Key Interfaces section code block and in Task 6.2. Added explicit explanation of how `evaluate()` calls `createEvaluationContext()` internally, and how the `Reference` data is fetched from `ReferencesStoragePart` using `findReferenceOrThrowException(ReferenceKey)` to obtain the reference's `AttributeValue[]` data.

- **`ExpressionVariableContext.getVariable()` return type** (Task 6.3): The WBS says `getVariable(String)` "returns the proxy from the map." However, the actual `ExpressionEvaluationContext` interface declares `getVariable()` as returning `@Nonnull Optional<Object>`. The implementation must return `Optional.ofNullable(map.get(variableName))`, not a raw object. The description should be more precise to avoid implementation confusion.

> ✅ **Addressed**: Task 6.3 now specifies that `getVariable(String)` returns `Optional.ofNullable(map.get(variableName))` matching the `@Nonnull Optional<Object>` return type, and that `getThis()` returns `Optional.empty()`.

- **`EntityContract` additional required methods not covered by any partial** (Tasks 2.1-2.7): `EntityContract` extends `ContentComparator<EntityContract>` (adding `differsFrom()`), `Versioned` (adding `version()`), and `Droppable` (adding `dropped()`). Task 2.7 covers `version()`, `dropped()`, `getScope()`, `getAllLocales()`, and `getLocales()`, but does not mention `differsFrom()`, `estimateSize()` (from `ContentComparator`), or several other methods from the full interface surface. The `CatchAllPartial` will handle these, but the WBS should acknowledge that `EntityVersionAndDroppablePartial` does not cover the full set of `Versioned`/`Droppable`/`ContentComparator` methods, so implementers know what falls through to `CatchAllPartial`.

> ✅ **Addressed**: Task 2.7 now includes a note explaining that `differsFrom()` and `estimateSize()` are default methods on `EntityContract` with working implementations (no explicit partial needed), that `exists()` from `Droppable` is also a default, and that any other unhandled methods fall through to `CatchAllPartial`.

- **Missing `AssociatedDataStoragePart` fetch granularity** (Task 4.1): The `StoragePartRecipe` declares `Set<AssociatedDataKey> neededAssociatedData` for fine-grained associated data fetching. However, the `EntityStoragePartAccessor.getAssociatedDataStoragePart()` method takes `(entityType, pk, AssociatedDataKey)` and returns a single `AssociatedDataStoragePart` per key. The `EntityProxyState` record (Task 1.1) declares `AssociatedDataStoragePart[] associatedDataParts` as an array. The WBS should clarify how the recipe's `Set<AssociatedDataKey>` is populated — the `AccessedDataFinder` produces paths like `$entity.associatedData['desc']`, which gives the name but not the locale. The `ExpressionProxyInstantiator` would need to know which locales to fetch. This is analogous to the attribute locale issue but is not discussed for associated data.

> ✅ **Addressed**: Task 4.1 now uses `Set<String> neededAssociatedDataNames` (names only from expression paths) and `Set<Locale> neededAssociatedDataLocales` (analogous to `neededAttributeLocales`). Added detailed explanation of how locale resolution works at trigger time: non-localized paths produce `AssociatedDataKey` with `locale=null`, while localized paths combine each name with locales from `neededAssociatedDataLocales` (or from `EntityBodyStoragePart.getLocales()` when "all locales" is indicated).

### Minor (nice to have)

- **`EntitySchemaPartial` handling of `getSchema()` return type**: `WithEntitySchema.getSchema()` returns `EntitySchemaContract`, but some callers might expect the more specific `SealedEntitySchema` subtype (since `SealedEntity` implementations typically return sealed schemas). The WBS correctly returns `EntitySchemaContract` from the state, but it would be worth noting that the proxy's `getSchema()` may return a non-sealed schema implementation, which differs from what `Entity.getSchema()` returns. This is unlikely to cause issues in the expression evaluation context, but it is a behavioral difference worth documenting.

> ✅ **Addressed**: Added a note to Task 2.1 explaining that the proxy's `getSchema()` may return a non-sealed schema implementation, and that this is acceptable in the expression evaluation context since accessors only use schema contract interface methods.

- **Thread safety mention missing**: The parent analysis mentions that `DataStoreMemoryBuffer` is transaction-scoped, but the WBS does not discuss thread safety of the proxy instances themselves. Since proxies are created and consumed within a single mutation processing thread, this is likely fine, but an explicit statement would be useful for future maintainers.

> ✅ **Addressed**: Added an explicit thread safety statement in the DataStoreMemoryBuffer caching section, confirming that proxy instances are created and consumed within a single mutation processing thread and require no synchronization.

- **Test for `unknown_variable_name_is_ignored_or_throws`** (PathToPartialMapperTest): The test description says "verify it is safely ignored (not mapped to any proxy)" but uses "or throws" phrasing. The behavior should be specified definitively — either it ignores unknown variables or it throws. Given that expressions are validated at schema load time, throwing seems more appropriate as a defensive measure.

> ✅ **Addressed**: Renamed test to `unknown_variable_name_throws_illegal_argument_exception` and updated description to specify that an unknown variable name throws `IllegalArgumentException` as a defensive fail-fast measure, since expressions are validated at schema load time.

- **`CatchAllPartial.OBJECT_METHODS` ordering**: Task 1.3 mentions that `CatchAllPartial` contains both `INSTANCE` (catch-all) and `OBJECT_METHODS` (for `toString`, `hashCode`, `equals`). The ordering is critical — `OBJECT_METHODS` must come before `INSTANCE` in the classification list. The WBS mentions this ("specific partials first, CatchAllPartial last") but does not explicitly state the internal ordering within `CatchAllPartial` itself. A clarifying note would help.

> ✅ **Addressed**: Task 1.3 now explicitly states that `OBJECT_METHODS` must appear before `INSTANCE` in the classification list, with the full ordering spelled out: all specific partials first, then `CatchAllPartial.OBJECT_METHODS`, then `CatchAllPartial.INSTANCE` last.

- **`ReferenceVersionAndDroppablePartial` version default value**: Task 3.6 sets `version=1`. The WBS should explain why this is a sensible default. If the reference data comes from `ReferencesStoragePart`, the actual reference version could be extracted from the `Reference` object. Using a hardcoded `1` might mask bugs in tests that verify version values.

> ✅ **Addressed**: Task 3.6 now reads the version from the `Reference` object (via `Reference.version()`) instead of using a hardcoded default. The `ReferenceProxyState` record (Task 1.2) was updated to include an `int version` field. The corresponding test case was also updated to verify the actual version from the reference entry.
