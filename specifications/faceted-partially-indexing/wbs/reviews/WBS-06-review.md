# WBS-06 Review: Target-Side Dispatch Infrastructure

## Verdict: PASS WITH NOTES

## Summary

WBS-06 is a well-organized document that clearly defines the three core abstractions (`IndexMutationTarget`, `IndexMutationExecutor<M>`, `IndexMutationExecutorRegistry`) and their collaboration pattern. The code examples are accurate, the delegation table for `EntityCollection implements IndexMutationTarget` is verified against the actual codebase, and the source code research section is exceptionally thorough. There are a few consistency issues with cross-references and one important gap in the `evaluateFilter` implementation guidance, but nothing that would block a competent developer from implementing this WBS.

## Issues Found

### Critical (must fix before implementation)

None.

### Important (should fix)

- **`IndexMutation` incorrectly called "sealed interface" in Dependencies section.** Line 39 states: "WBS-05 provides the `IndexMutation` sealed interface and `EntityIndexMutation` container type." WBS-05 consistently describes `IndexMutation` as a **marker interface**, not a sealed interface (WBS-05 lines 9, 20, 154, 284). The parent analysis document also declares it as `public interface IndexMutation extends MutationContract` with no `sealed` keyword. This mischaracterization could mislead a developer into adding a `sealed permits` clause to `IndexMutation`, which would be wrong since future mutation types (histogram etc.) need to implement it without modifying the base interface.

> ✅ **Addressed**: Changed "sealed interface" to "marker interface" on line 39 of the Dependencies section.

- **`evaluateFilter(FilterBy)` implementation guidance is insufficiently specific.** Group 2, task for `evaluateFilter` (lines 470-477) acknowledges this is "the most complex delegation" but does not commit to a concrete implementation approach. It lists two alternatives (construct `QueryPlanningContext` + `FilterByVisitor`, or use `FilterByVisitor.createFormulaForTheFilter()`) without recommending one. The source code research section (lines 415-423) identifies that `createFormulaForTheFilter` requires a `QueryPlanningContext`, `FilterBy`, `List<EntityIndex>`, and other parameters -- but does not resolve the key question of how to construct a minimal `EvitaRequest` and `QueryPlanningContext` without a real `EvitaSession`. This leaves a significant design decision to the implementer. A recommended approach (even if tentative) and a list of the exact constructor arguments needed would substantially reduce implementation ambiguity.

> ✅ **Addressed**: Rewrote the `evaluateFilter` task in Group 2 with a concrete recommended approach: use `FilterByVisitor.createFormulaForTheFilter()`, construct a minimal `EvitaRequest` via `EvitaRequest.fromQuery()`, create `QueryPlanningContext` via `EntityCollection.createQueryContext()`, and obtain a session via the catalog. Specific steps and constructor argument sources are now listed.

- **`IndexMutationExecutorRegistry` visibility mismatch between description and usage.** Group 4 (line 488) states the registry should be "package-private class (not public -- only accessed within `evita_engine`)", but it is in package `io.evitadb.index.mutation` while `EntityCollection` (which calls `IndexMutationExecutorRegistry.INSTANCE.dispatch(...)`) is in package `io.evitadb.core.collection`. Package-private access would make the class invisible to `EntityCollection`. The class needs to be `public` (or at minimum package-accessible to `EntityCollection`, which is not possible across different packages without `public`). The `dispatch()` method also needs to be `public` or package-accessible. The same issue applies to the `INSTANCE` field.

> ✅ **Addressed**: Changed the class to `public`, the `INSTANCE` field to `public static final`, and the `dispatch()` method to `public` — both in the task description (Group 4) and in the code example in the Technical Context section.

### Minor (nice to have)

- **`applyIndexMutations()` access modifier ambiguity.** Group 5 (line 501) notes the method "can be `package-private` (default access) if `LocalMutationExecutorCollector` is in the same package (`io.evitadb.core.collection`), or `public` if called from elsewhere." `LocalMutationExecutorCollector` is actually in `io.evitadb.index.mutation.storagePart` (confirmed from git status showing the file at `evita_engine/src/main/java/io/evitadb/index/mutation/storagePart/MutationCollector.java`), so the method would need to be `public`. A definitive recommendation would be clearer.

> ✅ **Addressed**: Replaced the ambiguous conditional ("can be package-private ... or public") with a definitive statement that the method must be `public`, citing the actual package of `LocalMutationExecutorCollector` (`io.evitadb.index.mutation.storagePart`).

- **Test class `IndexMutationTargetTest` includes overly prescriptive reflection-based contract tests.** Tests like `should_declare_exactly_six_methods` (line 519) and the individual `should_declare_*_with_nonnull_parameter_and_nonnull_return` tests (lines 521-526) test the interface's structural shape via reflection. While this verifies the contract, these tests are fragile -- any legitimate future addition of a 7th method would break them. They also duplicate what the compiler already enforces (method signatures, return types). The reflection tests for annotation presence (`@Nonnull`/`@Nullable`) are more useful since the compiler does not enforce annotation semantics, but the method-count and return-type tests add limited value over compilation checks. Consider whether the `EntityCollection` delegation tests (lines 530-538) alone provide sufficient contract coverage.

> ✅ **Addressed**: Replaced `should_declare_exactly_six_methods` with `should_declare_expected_methods` that checks for the presence of the 6 expected methods by name without asserting an exact count. Consolidated the 6 individual `should_declare_*_with_nonnull_parameter_and_nonnull_return` tests into a single `should_declare_nonnull_and_nullable_annotations_on_methods` test that focuses on annotation verification (valuable) and notes that return-type assertions are optional (compiler-enforced).

- **`ExpressionIndexTrigger` and `DependencyType` handling when not yet available.** Group 1 (line 459) and Group 4 (line 492) both mention using "temporary forward-reference stubs" or leaving code commented with `// TODO` references when downstream WBS types are unavailable. This is pragmatic, but the two groups handle the situation differently: Group 1 suggests "the interface can initially have 5 methods with `getTrigger()` added when its parameter types exist," while Group 4 suggests "use a temporary no-op executor stub or leave the map entry commented out." The document should adopt a consistent strategy. Given that WBS-03 defines `ExpressionIndexTrigger`/`DependencyType` and WBS-07 defines `ReevaluateFacetExpressionExecutor`, a clear "implementation order" note (e.g., "implement WBS-03 before WBS-06") would eliminate the need for stubs entirely.

> ✅ **Addressed**: Added a unified "Implementation order note" in Group 1 recommending "implement WBS-03 before WBS-06" to ensure types are available, eliminating the need for stubs. Updated Group 4's executor stub note to reference this same strategy ("consistent with the implementation order strategy described in Group 1") and to use a `// TODO: add when WBS-07 is implemented` comment format.

- **Test `INSTANCE_should_survive_entity_collection_transactional_copy` (line 552) is difficult to set up.** Creating an `EntityCollection` and calling `createCopyWithMergedTransactionalMemory()` requires significant infrastructure (catalog, persistence service, cache supervisor, etc.). The test's intent (verify the registry is a static singleton not affected by collection copies) is better validated by the simpler `registry_should_not_be_a_field_on_entity_collection` test (line 553) which achieves the same goal via reflection with far less setup complexity. Consider noting that this test may need to be an integration test rather than a unit test, or that the simpler reflection-based test is sufficient as a substitute.

> ✅ **Addressed**: Added a note to the test description recommending it be implemented as an integration test rather than a unit test, and noting that the simpler `registry_should_not_be_a_field_on_entity_collection` test provides equivalent validation with far less setup. Also enhanced the description of the simpler test to explain why it is the primary validation of the static singleton lifecycle.

- **`EntitySchema` vs `EntitySchemaContract` choice for `getEntitySchema()` return type could be documented more explicitly.** The source code research section (lines 442-444) correctly notes that `getInternalSchema()` returns `EntitySchema` (the concrete DTO class) and that the interface should match this. However, the rationale ("provides direct access to `ReferenceSchema` instances without casting") could mention that `EntitySchemaContract.getReference(String)` also returns `Optional<ReferenceSchemaContract>` without needing a cast. The real benefit is that `EntitySchema` provides the mutable/dto-specific internal view. This distinction matters for developers deciding whether to widen the return type in the future.

> ✅ **Addressed**: Expanded the rationale in the source code research section to acknowledge that `EntitySchemaContract.getReference(String)` also returns references without casting, and clarified that the real benefit of using `EntitySchema` is that it provides the concrete DTO/internal view needed by engine-level executors, not just the contract abstraction.

- **Handler hierarchy diagram (lines 265-283) duplicates AD 12 content.** The "Handler hierarchy diagram" section and the "AD 12 -- Registry-based executor dispatch" section (lines 289-294) present essentially the same information in different formats (ASCII diagram vs. numbered prose). While the duplication aids readability, it creates a maintenance burden. Consider cross-referencing one from the other rather than repeating.

> ✅ **Addressed**: Added a cross-reference note to the diagram heading ("visual summary -- see AD 12 below for detailed prose description") to make the relationship between the diagram and AD 12 explicit, reducing maintenance burden while preserving both formats for readability.
