# WBS-03 Review: Expression Index Trigger

## Verdict: PASS WITH NOTES

## Summary

WBS-03 is a well-structured and thorough specification for the trigger abstraction layer. The interface definitions, construction pipeline, and evaluation modes are clearly described with sufficient detail for implementation. There are a few inconsistencies (most notably the `entityGroupHaving` vs `groupHaving` naming), one unresolved design decision that should be settled before implementation, and some gaps in test coverage for error/edge cases. None of these are blocking, but several should be addressed to avoid implementation rework.

## Issues Found

### Critical (must fix before implementation)

- **Unresolved design decision in Task 5.3.** The task explicitly states "Design decision needed: document the chosen approach" for how local-only expressions (no cross-entity paths) should be handled. The two options are: (a) build a `FacetExpressionTrigger` with a sentinel/null `DependencyType`, or (b) skip trigger construction entirely and let `ReferenceIndexMutator` handle it inline. This decision has downstream implications for the interface contract (`getDependencyType()` is declared `@Nonnull`) and for WBS-09 (local trigger integration). If option (a) is chosen, either `DependencyType` needs a `LOCAL` value or `getDependencyType()` must become `@Nullable`, both of which ripple into WBS-04 and WBS-08. This must be resolved before implementation begins.

> **Addressed**: Resolved Task 5.3 with a hybrid approach: the factory builds a `FacetExpressionTrigger` for local-only expressions (needed for `evaluate()` in `ReferenceIndexMutator`), but with `getDependencyType()` returning `null`, `getDependentAttributes()` returning empty set, and `getFilterByConstraint()` throwing `UnsupportedOperationException`. The interface's `getDependencyType()` was changed from `@Nonnull` to `@Nullable`. Task 7.1 updated to skip cross-entity registry registration for triggers with null dependency type. The decision and all downstream implications are now documented inline.

- **`entityGroupHaving` vs `groupHaving` naming inconsistency within WBS-03.** The document itself acknowledges in the "Key design decisions" section (task area item 3) that WBS-01 corrects `entityGroupHaving` to `groupHaving` -- the actual evitaDB constraint is `QueryConstraints.groupHaving()` / `GroupHaving`. However, the `ExpressionIndexTrigger` interface JavaDoc (line 172) and the cross-entity evaluation notes (line 418) still use the incorrect `entityGroupHaving` name. Since this is interface-level JavaDoc that developers will copy-paste or reference during implementation, the wrong name will propagate into code. All occurrences of `entityGroupHaving` in WBS-03 should be updated to `groupHaving` to match the actual API. (Note: the parent analysis document also uses `entityGroupHaving` throughout, but WBS-03 should reflect the corrected name per its own stated correction.)

> **Addressed**: Replaced all occurrences of `entityGroupHaving(entityPrimaryKeyInSet(mutatedPK))` with `groupHaving(entityPrimaryKeyInSet(mutatedPK))` throughout WBS-03 (interface JavaDoc at line 172 and cross-entity evaluation notes at line 418). The only remaining mention of `entityGroupHaving` is in the key design decisions note that explains the correction itself (item 3: "WBS-01 corrects `entityGroupHaving` -> `groupHaving`").

### Important (should fix)

- **No error handling specification for `evaluate()` when expression returns non-boolean.** The `evaluate()` method is specified to "compute the expression result and cast to boolean." What happens if the expression evaluates to `null`, an integer, or a string? The parent analysis does not address this either. The WBS should specify the expected behavior: throw `ExpressionEvaluationException`? Treat `null` as `false`? This is not theoretical -- a misconfigured expression like `$entity.attributes['name']` (returns a String, not boolean) would hit this path. At minimum, add an acceptance criterion and a test case for non-boolean expression results.

> **Addressed**: Added explicit non-boolean result handling specification to both the "Trigger Time Evaluation" section (step 5) and Task 4.2 (step 6): `Boolean` used directly, `null` treated as `false`, any other type throws `ExpressionEvaluationException`. Added four new test cases in Test Class 3 under a new "non-boolean and error results" category: `should_treat_null_expression_result_as_false`, `should_throw_for_non_boolean_expression_result`, `should_throw_for_integer_expression_result`, and `should_propagate_ExpressionEvaluationException_from_catch_all_partial`.

- **`getDependencyType()` contract tension with local-only triggers.** Even setting aside the unresolved Task 5.3 decision, there is a structural issue: the `ExpressionIndexTrigger` interface declares `@Nonnull DependencyType getDependencyType()`, but `DependencyType` only has two values (`REFERENCED_ENTITY_ATTRIBUTE`, `GROUP_ENTITY_ATTRIBUTE`), both of which represent cross-entity relationships. If the trigger is used for local evaluation (which the document explicitly states it is), what `DependencyType` should a local-only trigger return? The enum JavaDoc says "Only cross-entity relationships are represented here." This tension should be resolved in the interface design, not left to implementers.

> **Addressed**: Changed `getDependencyType()` from `@Nonnull` to `@Nullable` in both the interface JavaDoc (code block) and Task 2.1. Local-only triggers return `null`, which is consistent with the enum JavaDoc stating only cross-entity relationships are represented. The `DependencyType` enum remains unchanged (no `LOCAL` value needed). Updated the reflective test in Test Class 2 to verify `@Nullable` annotation.

- **Missing test coverage for expression evaluation failure modes.** The test plan covers true/false results, null attributes, and storage part interaction, but does not include tests for: (a) expression throwing an exception during evaluation (e.g., `ExpressionEvaluationException` from the CatchAllPartial), (b) `storageAccessor` returning null for a required storage part, (c) expression referencing an attribute that does not exist in the schema. These are realistic error paths that should have explicit test coverage.

> **Addressed**: Added six new test cases across two new categories in Test Class 3: (a) `should_propagate_ExpressionEvaluationException_from_catch_all_partial` covers CatchAllPartial exceptions; (b) `should_throw_when_required_storage_part_is_missing` and `should_handle_missing_referenced_entity_storage_parts_gracefully` cover storage accessor failure modes; (c) non-boolean expression result tests (`should_throw_for_non_boolean_expression_result`, `should_throw_for_integer_expression_result`) cover misconfigured expressions. The "attribute not in schema" case is implicitly covered by the CatchAllPartial test (accessing unrecognized methods).

- **Task 5.2 step (b) says "create one trigger per dependency type" for mixed cross-entity expressions, but Task 5.4 says the `DependencyType` is "determined by the cross-entity portion."** These two statements cover different cases (5.2b: expression depends on BOTH group and referenced entity; 5.4: expression mixes cross-entity with local paths), but the boundary between them is not explicit. A reader could interpret 5.4 as contradicting 5.2b for an expression like `$reference.groupEntity?.attributes['a'] && $reference.referencedEntity.attributes['b'] && $entity.attributes['c']`. The WBS should clarify: this expression produces two triggers (per 5.2b), each with its own `dependentAttributes` set (per 5.2c), and both carry the same full `FilterBy` (per 5.4's "single FilterBy tree" principle).

> **Addressed**: Expanded Task 5.4 with an explicit clarification of the boundary between 5.2b and 5.4. The text now includes a concrete example showing that an expression like `$reference.groupEntity?.attributes['a'] && $reference.referencedEntity.attributes['b'] && $entity.attributes['c']` produces two triggers (per 5.2b), each with its own `dependentAttributes` set, both carrying the full `FilterBy` tree. The roles of 5.2b (multi-source cross-entity) and 5.4 (cross-entity + local mix) are explicitly delineated.

- **No specification of `FilterBy` content for triggers with ONLY local dependencies.** If a trigger is created for a purely local expression (per one possible resolution of Task 5.3), what does `getFilterByConstraint()` return? The `FilterBy` translation is designed for cross-entity evaluation. For a purely local expression like `$entity.attributes['isActive'] == true`, the translated `FilterBy` would be `filterBy(attributeEquals("isActive", true))` -- valid but not scoped to any reference. The document does not discuss whether this is correct or even useful for local-only triggers.

> **Addressed**: Resolved by the Task 5.3 design decision: local-only triggers have `getFilterByConstraint()` throw `UnsupportedOperationException` (no `FilterBy` is built or stored). The cross-entity evaluation section now includes an explicit note that `getFilterByConstraint()` is only available on triggers with non-null `getDependencyType()`. Local-only expressions are evaluated exclusively via `evaluate()` inline in `ReferenceIndexMutator`.

### Minor (nice to have)

- **Test Class 2 (`ExpressionIndexTriggerContractTest`) is mostly reflective type-checking.** Tests like "verify method exists via reflection and has `@Nonnull` annotation" add minimal value -- the compiler already enforces method signatures, and annotation presence is a style concern. These tests will become maintenance burden if the interface evolves. Consider whether a few of these could be replaced by compile-time checks or simply omitted in favor of behavioral tests.

> **Declined**: While the observation is valid that these tests add limited value beyond what the compiler enforces, they serve as documentation-as-tests for the interface contract and are low-cost to maintain. The evitaDB codebase has a 70% line coverage requirement and these tests contribute to that. The behavioral tests in Test Class 3 are the primary coverage for the trigger; these reflective tests are a secondary safety net. No changes made.

- **Ordinal stability tests for `DependencyType` (Test Class 1) imply serialization.** The tests verify `REFERENCED_ENTITY_ATTRIBUTE.ordinal() == 0` "to guard against accidental reordering that could break serialized data." However, the parent analysis explicitly states that `IndexMutation` instances are "never written to WAL" and "never serialized." If `DependencyType` is never persisted, ordinal stability tests are unnecessary. If there IS a serialization concern not documented, it should be made explicit.

> **Addressed**: Updated the ordinal stability test descriptions to note that `DependencyType` is not currently serialized and the tests are defensive measures against future changes, not serialization correctness requirements. The misleading "could break serialized data" justification has been replaced.

- **The `FacetExpressionTriggerFactory` naming.** The document introduces a factory class (`FacetExpressionTriggerFactory`) in Task Group 5 but then also mentions a `buildAndRegisterTriggers(EntitySchema)` method in Task Group 7. The relationship between these two is clear (7.1 calls 5.2), but the naming suggests the factory both builds and registers, while the code in 7.1 does the registration. Consider clarifying that the factory only builds, and registration is a separate concern.

> **Addressed**: Updated Task 5.1 description to explicitly state that the factory "builds (but does NOT register)" triggers, and that "Registration in `CatalogExpressionTriggerRegistry` is a separate concern handled by the caller (Task 7.1's `buildAndRegisterTriggers` method)."

- **Test Class 6 (`TriggerEvaluationCachingTest`) relies on caching behavior of `DataStoreMemoryBuffer`.** The parent analysis notes "To be verified by integration tests: Confirm that `DataStoreMemoryBuffer` makes entity B's pending changes visible to entity A's accessor." This WBS's tests assume the buffer provides caching, but the caching verification itself is not within WBS-03's scope. The tests should note this dependency or be conditional on the buffer integration test passing.

> **Addressed**: Added a dependency note block above the Test Class 6 caching tests, explicitly documenting that these tests depend on `DataStoreMemoryBuffer`'s caching behavior and cross-referencing the parent analysis's integration test verification requirement.

- **AD-12 and AD-13 references lack context.** The document references "AD-12" and "AD-13" in section headers ("AD-12 Context: Registry-Based Executor Dispatch", "AD-13 Context: Cross-Schema Wiring") without explaining what these are or where they are defined. If these are architecture decision records from the parent analysis, they should be cross-referenced with section names or line numbers.

> **Addressed**: Added blockquote cross-references under both the "AD-12 Context" and "AD-13 Context" section headers, linking to the parent analysis document's "Architecture Decisions" section with the full decision name (e.g., "Architecture Decision 12 â€” Registry-based executor dispatch â€” stateless static singleton").
